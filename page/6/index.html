HTTPS 工作流程详解

1. TCP 握手阶段

  • TCP SYN: 客户端向服务器发送同步请求，请求建立连接
  • TCP SYN + ACK: 服务器响应并确认连接请求
  • TCP ACK: 客户端确认，TCP 连接建立完成

这是标准的 TCP 三次握手，为后续的 TLS/SSL 通信建立可靠的传输层连接。

2. 证书验证阶段

  • Client Hello: 客户端发送支持的 TLS 版本、加密套件列表等信息
  • Server Hello: 服务器选择 TLS 版本和加密套件
  • Certificate: 服务器发送数字证书（包含公钥）
  • Server Hello Done: 服务器表示握手信息发送完毕

在这个阶段，客户端会验证服务器证书的有效性，包括：

  • 证书是否由可信 CA 签发
  • 证书是否在有效期内
  • 证书中的域名是否与访问的域名匹配

3. 密钥交换阶段

这是 HTTPS 安全性的核心部分：

非对称加密过程：

  • 客户端生成一个随机的 session key（会话密钥）
  • 使用服务器证书中的 公钥 对 session key 进行加密
  • 发送 Client Key Exchange 消息，包含加密后的 session key
  • 服务器使用自己的 私钥 解密获得 session key

握手完成：

  • Change Cipher Spec: 双方通知对方后续通信将使用协商好的加密参数
  • Finished: 使用新的加密参数发送握手完成消息

4. 数据传输阶段

对称加密通信：

  • 双方都拥有相同的 session key
  • 使用对称加密算法（如 AES）对数据进行加密/解密
  • 所有 HTTP 数据都通过这个加密通道传输

加密机制说明

为什么使用混合加密？

  1. 非对称加密（RSA/ECDH）：

    • 优点：密钥分发安全，无需预共享密钥
    • 缺点：计算开销大，速度慢

  2. 对称加密（AES）：

    • 优点：加密速度快，效率高
    • 缺点：密钥分发困难

HTTPS 的巧妙之处：

  • 用非对称加密安全地交换对称密钥
  • 用对称加密进行实际的数据传输
  • 既保证了安全性，又保证了效率

7. 安全检查清单

  • ✅ 使用 TLS 1.2 或更高版本
  • ✅ 禁用不安全的加密套件
  • ✅ 启用 HSTS
  • ✅ 配置正确的证书链
  • ✅ 定期更新证书
  • ✅ 监控证书过期时间
  • ✅ 使用强加密算法（AES-256）
  • ✅ 启用 Perfect Forward Secrecy

🧭 PostgreSQL 与 MySQL 对比表（操作/概念）

类别 PostgreSQL MySQL 差异说明
CLI 工具 psql mysql 使用方式相似，但命令格式略有不同
查看数据库 \l SHOW DATABASES; PostgreSQL 使用 \ 开头的元命令
查看表 \dt SHOW TABLES; MySQL 更偏向 SQL 语句
查看建表语句 pg_dump -s -t 表名 或 \d+ 表名 SHOW CREATE TABLE 表名; PostgreSQL 没有直接 SHOW CREATE 语法
切换数据库 \c USE 数据库名; 语法不同
分隔符设置 \pset format 无等价命令 PostgreSQL CLI 更强大
默认字符集 UTF-8 latin1（可改） PostgreSQL 默认更现代
JSON 支持 强，原生 JSON、JSONB 类型 有 JSON 类型，但功能弱很多 PostgreSQL 更适合结构+非结构混合存储
权限模型 支持角色/继承/细粒度权限 用户权限简单 PostgreSQL 适合大型多租户
存储引擎 只有 heap（可选 WAL 日志） 多引擎（默认 InnoDB） MySQL 更灵活但容易不一致

单元测试的标准与最佳实践

测试模式分类

1. 表驱动测试（table-driven test）

  • 是 Go 语言社区非常推崇的一种测试模式。它的核心思想是：
  • 把一组输入和期望输出写成一个“表”（通常是切片或数组），然后用循环依次执行每组用例。

表驱动测试的基本结构

适用场景

  • 输入输出明确、参数组合多的函数
  • 需要覆盖多种边界情况的逻辑

2.子测试（Subtests）

  • 简介：使用 t.Run 创建子测试，便于分组、并发和独立执行。
  • 优点：可以单独运行某个子测试，输出更清晰。
  • 示例：

3.基准测试（Benchmark）

  • 简介：以 Benchmark 开头的函数，用于性能测试。
  • 规范：函数签名为 func(b *testing.B)，通过循环多次执行被测代码。
  • 示例：

4.并发测试（Parallel Test）

  • 简介：使用 t.Parallel() 让测试用例并发执行，检测并发安全问题。

  • 示例：

5. 测试覆盖率（Coverage）

  • 简介：通过 go test -cover 查看测试覆盖率，确保代码质量。
  • 建议：覆盖率不是越高越好，但应覆盖核心逻辑和边界情况。

6. 断言库（Assertion Library）

  • 简介：使用断言库简化判断和输出，如 github.com/stretchr/testify/assert。
  • 示例：

7. Mock 和 Stub

  • 简介：通过接口和自定义实现，模拟外部依赖（如数据库、网络等），实现隔离测试。
  • 常用库：github.com/stretchr/testify/mock、golang/mock。

8. 测试文件规范

  • 测试文件以 _test.go 结尾。
  • 测试函数以 Test、Benchmark、Example 开头。
  • 测试代码和生产代码分离。

controller工作机制

发表于 2025-05-26

Kubernetes Controller 工作机制解析

https://github.com/kubernetes/client-go/blob/master/examples/workqueue/main.go

控制器核心组件

1. 初始化配置 (Configuration)

  • 通过kubeconfig文件或in-cluster配置初始化与API Server的连接
  • 创建clientset用于与Kubernetes API交互

2. 资源监听器设置 (Resource Watcher)

  • 创建ListWatch，指定要监听的资源类型(如Pod)和范围(如命名空间)
  • ListWatch负责与API Server建立连接，监听资源变化

3. 工作队列创建 (Workqueue)

  • 创建RateLimitingQueue作为工作队列
  • 队列支持限速重试，防止资源过度消耗
  • 确保并发安全的事件处理

4. 缓存和事件处理器配置 (Cache & Informer)

  • 创建Indexer作为本地缓存，存储资源对象
  • 创建Informer监听资源变化并更新本地缓存
  • 设置事件处理函数(Add/Update/Delete)，将变化的资源键加入工作队列

5. 控制器组装 (Controller Assembly)

  • 将队列(queue)、缓存(indexer)和监听器(informer)组装成控制器
  • 形成完整的事件处理流水线
  • 控制器负责协调这些组件的工作

6. 业务逻辑处理 (Business Logic)

  • processNextItem不断从队列获取待处理项
  • 通过indexer从本地缓存获取最新的资源状态
  • 执行具体的业务逻辑(如本例中的打印资源信息)
  • 处理完成后从队列中移除，失败则重新入队

详细工作流程

流程说明

  1. Informer与API Server的交互

    • 首先通过List获取资源的完整列表
    • 然后通过Watch监听资源的变化事件

  2. Informer更新本地缓存

    • 收到List响应时，将资源列表存入Indexer
    • 收到Watch事件时，相应更新Indexer中的数据
    • Indexer始终保持与服务器端数据的同步

  3. 事件处理流程

    • 当收到事件时，调用相应的EventHandler(Add/Update/Delete)
    • EventHandler将资源的key放入WorkQueue
    • EventHandler不直接操作Indexer

  4. 控制器业务处理

    • 从WorkQueue取出要处理的资源key
    • 通过key从Indexer(本地缓存)中获取资源的最新状态
    • 执行具体的业务逻辑

设计优点

  1. 数据一致性

    • Indexer的更新由Informer直接负责
    • 确保缓存与服务器端的一致性

  2. 职责分离

    • EventHandler只负责将事件对应的key放入队列
    • 各组件职责单一，界限清晰

  3. 性能优化

    • 业务逻辑通过Indexer读取数据
    • 避免频繁访问API Server

  4. 可靠性

    • 支持失败重试和限速机制
    • 确保资源状态的最终一致性
