路由Demo解析

发表于 2026-01-27
本文字数： 阅读时长 ≈ NaN:aN

时序流程图（Mermaid）

一、“总纲式结论”

跨三层通信的本质是：

  • IP 地址：端到端不变（逻辑寻址）
  • MAC 地址：逐跳改变（物理转发）
  • ARP：只解决“下一跳是谁”，而不是“最终是谁”

二、拓扑与角色明确

拓扑如下（你给的）：

  • Host1 的默认网关：Router A
  • Host2 直连在 Router C 的 LAN 上
  • 路由器之间都是三层转发
  • 终端和路由器之间是二层通信

三、完整路由过程（按步骤拆解）

阶段 1：Host1 决定“我该把包交给谁”

Step 1：Host1 判断目的 IP 是否本地

  • Host1 拿到 Host2 的 IP

  • 用自己的：

    • IP
    • 子网掩码

  • 计算后发现：
    Host2 不在本地子网

👉 结论：必须交给默认网关（Router A）

Step 2：Host1 通过 ARP 找默认网关的 MAC

  • 查看 ARP 表：

    • 有 Router A MAC → 直接用
    • 没有 → ARP Broadcast

  • 得到：

    • Router A（LAN 口）的 MAC

Step 3：Host1 封装第一个以太网帧

此时数据包状态是：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Host1 MAC
Dest MAC Router A（LAN 口）MAC

👉 关键点：
Host1 不知道 Router B / C / Host2 的 MAC，也不需要知道。

阶段 2：Router A → Router B

Step 4：Router A 收到帧（L2 → L3）

Router A 做三件事：

  1. 看 目的 MAC → 是自己 → 收包

  2. 剥掉二层头，检查 目的 IP

  3. 查路由表：

    • Host2 不直连
    • 找到下一跳：Router B

Step 5：Router A 通过 ARP 找 Router B 的 MAC

  • ARP 查询：

    • IP：Router B（A-B 链路上的 IP）
    • 得到 Router B 接口 MAC

Step 6：Router A 重新封装帧并转发

新的帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router A（发往 B 的接口 MAC）
Dest MAC Router B（面向 A 的接口 MAC）

👉 IP 没变，MAC 换了

阶段 3：Router B → Router C（同理）

Router B 做的事情和 Router A 完全一样：

  1. 收到帧（MAC 是自己）
  2. 解封装
  3. 查路由表
  4. 下一跳是 Router C
  5. ARP 找 Router C MAC
  6. 重新封装并转发

新的帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router B（面向 C 的接口 MAC）
Dest MAC Router C（面向 B 的接口 MAC）

阶段 4：Router C → Host2（最后一跳）

Step 7：Router C 发现目的 IP 是本地直连

  • 查路由表：

    • Host2 IP 在本地 LAN

  • 不再需要“下一跳路由器”

Step 8：Router C ARP 查询 Host2 的 MAC

  • ARP：

    • IP：Host2
    • 得到 Host2 MAC

Step 9：Router C 发送最终帧

最终帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router C（LAN 口 MAC）
Dest MAC Host2 MAC

Step 10：交换机（二层）转发

  • 交换机只看 Destination MAC
  • 查 MAC 表
  • 发给 Host2 所在端口

👉 Host2 收到数据，去程完成

四、一路上哪些东西「一直不变」，哪些「一直在变」

✅ 一直不变的

  • Source IP：Host1
  • Destination IP：Host2

🔄 每一跳都会变的

  • Source MAC
  • Destination MAC

🧠 决策逻辑

  • IP 决定“去哪里”
  • MAC 决定“下一跳交给谁”

五、为什么不能“全网 ARP”（你原文最后的点睛）

如果没有路由器：

  • 所有设备在一个广播域
  • 每一次通信都 ARP 全网
  • 广播风暴
  • 网络直接瘫痪

👉 路由器的核心价值：

  • 切割广播域
  • 限制 ARP 的作用范围
  • 让 Internet 成为可能

六、一句“工程师级总结”

三层决定方向，二层负责搬运。
路由器每一跳只关心：
“下一步我该把包交给谁？”

一、分析图片的 3 个步骤

Step 1：给 Pod 插上“网线”（veth）

创建 veth pair，一端进 Pod，一端留在 Node

👍 这是 CNI 的核心职责之一

底层原理：

  • veth 是 Linux 提供的“虚拟网线”
  • Pod 里看到的是 eth0
  • Node 上看到的是 vethxxxx
    👉 CNI 插件最基础的职责：

    把 Pod 的 network namespace 接入宿主机网络

Step 2：给 Pod 分配 IP（IPAM）

每个 Pod 一个唯一 IP，通常 Node 拿一个子网

👍 这也是 CNI 的标准设计

这是：

  • Node 级子网划分
  • Pod 从 Node 子网里分 IP
    👉 这是非常典型的 Node-scoped IPAM 模型
    Flannel / Calico / Cilium 都支持这个模型

Step 3：配置 IP + 路由（L3 通）

Pod 能出网，Node 知道怎么把包送回 Pod

👍 这是最容易被低估，但最重要的一步

图里提到三件事：

  1. 给 Pod eth0 配 IP
  2. 给 Pod 配默认路由
  3. 给 Node 配到 Pod IP 的路由
    这是 “Pod 能通信”成立的最小条件。

发表于 2026-01-20

一、TCP 半连接 / 全连接核心参数对照表

1️⃣ net.ipv4.tcp_max_syn_backlog

项目 内容
名称 tcp_max_syn_backlog
作用阶段 半连接队列（SYN_RECV 阶段）
含义 内核允许的最大 半连接（SYN 已收到，三次握手未完成） 数量
典型现象 队列满时：不回 SYN+ACK 或启用 syncookies
你的当前值 4096
不同值的影响
  • 小：突发连接时大量 SYN 超时
  • 大：提升并发连接建立能力（需内存）
查看方式 cat /proc/sys/net/ipv4/tcp_max_syn_backlog
是否单独生效 ❌ 否（受 somaxconn、listen backlog 共同限制）

2️⃣ net.ipv4.tcp_syncookies

项目 内容
名称 tcp_syncookies
作用阶段 半连接队列溢出时
含义 半连接队列满时，是否启用 SYN Cookies
你的当前值 1
不同值的作用
  • 0：关闭（队列满直接丢 SYN）
  • 1：仅队列满时启用（⭐推荐）
  • 2：无条件启用（防 SYN flood）
查看方式 cat /proc/sys/net/ipv4/tcp_syncookies
注意点 开启时部分 TCP 扩展（窗口扩大、时间戳）可能受限

✅ 结论：你的配置是生产环境最合理值

3️⃣ net.core.somaxconn

项目 内容
名称 somaxconn
作用阶段 全连接队列（accept queue）
含义 系统层面允许的 listen backlog 最大值
你的当前值 4096
不同值的影响
  • 小：accept 队列很容易满
  • 大：支撑高并发 accept
查看方式 cat /proc/sys/net/core/somaxconn
是否决定最终值 ❌ 否（还取决于应用 listen backlog）

4️⃣ net.ipv4.tcp_abort_on_overflow

项目 内容
名称 tcp_abort_on_overflow
作用阶段 全连接队列满
含义 accept 队列满时，如何处理客户端 ACK
你的当前值 0
不同值的作用
  • 0：丢 ACK（客户端重试 → 看起来像超时）
  • 1：回 RST（客户端立刻失败）
查看方式 cat /proc/sys/net/ipv4/tcp_abort_on_overflow
适用场景
  • 0：线上业务（避免误伤）
  • 1：压测/定位问题

⚠️ 这是排障神器，但不建议长期线上开启 1

5️⃣ net.ipv4.tcp_synack_retries

项目 内容
名称 tcp_synack_retries
作用阶段 半连接保活
含义 SYN+ACK 最大重传次数
你的当前值 5
默认值 通常是 5
不同值的影响
  • 大：半连接驻留时间长
  • 小：更快释放半连接
查看方式 cat /proc/sys/net/ipv4/tcp_synack_retries
经验值 2～5（内网服务可适当调小）

ps -eo 常用字段速查（最佳实践）

1️⃣ 调度/性能诊断必备组合

🔹 看调度类 + 优先级

用途：

  • 区分 CFS / RT / DL
  • 判断 RT 是否抢占系统

🔹 看 nice 值

用途：

  • 分析 CFS 内部竞争
  • 判断是否需要 renice

🔹 看 CPU 亲和性（配合 taskset）

字段 含义
psr 当前运行 CPU

🔹 看上下文切换压力

字段 含义
cswch 自愿上下文切换
nvcswch 被动抢占

🔹 看调度延迟相关（高级）

  • D 状态频繁 → IO / 调度阻塞
  • R 但运行慢 → 被抢占

2️⃣ 一条「万能调度观察命令」

📌 建议你直接 alias 掉

chrt：调度策略的“手术刀”

chrt = change / show real-time scheduling attributes

1️⃣ 查看进程调度策略

示例输出：

说明：

  • SCHED_OTHER = CFS
  • priority=0 对 CFS 没意义（RT 才用）

2️⃣ 修改为 RT（⚠️ 慎用）

🔹 改为 FIFO（强实时）

  • -f = SCHED_FIFO
  • 20 = RT priority（1–99）

⚠️ 风险极高：

  • 不主动 yield 会霸占 CPU
  • 可直接卡死系统

🔹 改为 RR（可轮转）

  • 更安全
  • 适合周期性内核线程（如网络）

3️⃣ 只“稍微扶一把”：nice 而不是 RT

🔹 推荐方案（ksoftirqd 场景）

效果：

  • 仍是 CFS
  • 但 vruntime 增长更慢
  • 显著降低被饿死概率

👉 这是生产环境首选

4️⃣ 一次性批量调整 ksoftirqd

📌 不建议 < -10

调度相关的其他关键观测工具

适合：

  • NUMA / 大核系统
  • 分析 runqueue 压力

3️⃣ perf sched

用途：

  • 精确看到谁在抢 CPU
  • 定位调度延迟根因

4️⃣ eBPF（你已经在用的最高阶）

  • sched:sched_switch
  • irq:softirq_raise
  • irq:softirq_entry

👉 你现在做的事情，已经是调度问题的最深水区

五、结合你当前问题的结论性建议

✅ 关于 ksoftirqd：

  • 不是随机 CPU

  • 是 per-cpu 绑定线程

  • 被饿死 ≠ 优先级低，而是：

    • CFS 公平性
    • 与业务进程同级竞争

✅ 推荐干预顺序（控制变量）：

  1. ❌ 不启 RPS
  2. ✅ renice -5 ksoftirqd/*
  3. ✅ IRQ 绑核（同 NUMA）
  4. ⛔ 谨慎使用 chrt

IRQBALANCE_BANNED_CPUS 掩码含义详解

基本概念

IRQBALANCE_BANNED_CPUS 是 IRQ Balance 服务的配置参数，用于指定禁止IRQ Balance 自动分配中断的 CPU 核心。

位值含义

关键理解：

  • 0 bit = 允许使用 - IRQ Balance 可以将中断分配到这个 CPU
  • 1 bit = 禁止使用 - IRQ Balance 不会将中断分配到这个 CPU

示例说明

假设一个 8 核系统 (CPU 0-7)：

转换为十六进制: 4c

这意味着：

  • CPU 0,1,4,5,7 - 允许 IRQ Balance 分配中断 (bit=0)
  • CPU 2,3,6 - 禁止 IRQ Balance 分配中断 (bit=1)

实际应用场景

1. 网卡中断优化

2. NUMA 优化

3. 应用程序 CPU 隔离

我们脚本中的逻辑

在优化后的 irqbalance.sh 脚本中：

脚本目标

  • 根据网卡所在的 NUMA 节点，选择一部分 CPU 核心专门处理网卡中断
  • 将其余 CPU 核心标记为”禁止”，让它们专注于应用程序处理

处理流程

  1. 获取允许的 CPU (ALLOWED_CPUS) - 网卡可以使用的 CPU
  2. 计算禁止的 CPU (BANNED_CPUS) - 其余所有 CPU
  3. 生成掩码 - 将 BANNED_CPUS 对应的位设置为 1
