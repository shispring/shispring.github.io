调度注意

k8s 原生调度器架构

  • Scheduling Cycle 是串行的，Binding Cycle 是并行的
  • Filter 失败 → FeasibleNodes 为空 → 怎么处理
  • Score 打分 → NormalizeScore → 选最高分节点

Scheduler Framework 扩展点

  • Filter：过滤不可调度节点（资源不足、亲和性、污点等）
  • Score：对通过 Filter 的节点打分（LeastAllocated / MostAllocated / ImageLocality…）
  • Permit：可以 Wait，这是 Gang Scheduling 的关键 hook（Volcano/Coscheduling 都挂这里）
  • Reserve：乐观占位，失败回滚
  • Bind：写 pod.spec.nodeName

Volcano 核心概念

概念 作用
Queue 资源配额隔离 + 公平调度单元
PodGroup Gang 的最小单元，替代原生 Pod 粒度
vcjob 高层封装，自动创建 PodGroup
Action Allocate / Preempt / Reclaim / Backfill（Volcano 调度循环）

核心调度插件 核心作用（精准释义）
Gang Scheduling（组队调度） 采用「全有或全无」调度机制，训练任务所有 Pod（PS/Worker）资源全部满足后才统一启动，杜绝部分 Pod 就绪、部分 Pending 的卡死死锁问题，完美适配分布式训练场景。
Queue（队列优先级调度） 基于自定义队列划分资源池，支持权重、资源配额、抢占保护配置，高优先级业务优先调度，可抢占低优先级空闲任务资源，保障核心AI训练任务优先运行。
Fair-Share / DRF（公平资源调度） 多团队/多队列共享GPU集群时，按队列权重动态分配空闲资源，基于DRF主导资源公平算法均衡资源占比，避免单业务、单团队独占集群GPU/CPU资源，实现资源合理均分。
Bin Packing / Spread（节点排布调度） 双策略灵活适配场景：Bin Packing 紧凑打包调度，填满节点资源、减少资源碎片，提升集群资源利用率；Spread 分散打散调度，将Pod分布在不同节点，规避单点故障，提升任务容错性。
Topology Aware（拓扑感知调度） 硬件层级感知集群拓扑，自动识别NVLink、RDMA、网卡、机架等网络层级，优先将通信频繁的训练Pod调度至近距离节点，大幅降低多卡/多机分布式训练的通信延迟。

调度算法

  • DRF（Dominant Resource Fairness）：多维资源公平，必须会推导
  • Bin-Packing vs Spread：GPU 场景偏 bin-packing（节省碎片），在线服务偏 spread（高可用）
  • Backfill：用小作业填空隙，提高利用率
  • Preemption vs Reclaim：preempt 抢跨 Queue 的低优先级；reclaim 拿回本 Queue 被借出的资源

调度架构

维度 PS-Worker（Volcano两类task） AllReduce 架构（无PS）
节点角色 PS（存参数）+ Worker（算梯度） 全部都是Worker，无独立参数节点
梯度聚合 PS单点汇总所有梯度 Worker之间互相AllReduce交换梯度
Volcano定义 tasks分ps、worker两套模板 只定义单一worker task
适用场景 超大参数量模型、参数分片 中小模型、多卡多机高速互联
