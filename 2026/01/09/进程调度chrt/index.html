ps -eo 常用字段速查（最佳实践）

1️⃣ 调度/性能诊断必备组合

🔹 看调度类 + 优先级

用途：

  • 区分 CFS / RT / DL
  • 判断 RT 是否抢占系统

🔹 看 nice 值

用途：

  • 分析 CFS 内部竞争
  • 判断是否需要 renice

🔹 看 CPU 亲和性（配合 taskset）

字段 含义
psr 当前运行 CPU

🔹 看上下文切换压力

字段 含义
cswch 自愿上下文切换
nvcswch 被动抢占

🔹 看调度延迟相关（高级）

  • D 状态频繁 → IO / 调度阻塞
  • R 但运行慢 → 被抢占

2️⃣ 一条「万能调度观察命令」

📌 建议你直接 alias 掉

chrt：调度策略的“手术刀”

chrt = change / show real-time scheduling attributes

1️⃣ 查看进程调度策略

示例输出：

说明：

  • SCHED_OTHER = CFS
  • priority=0 对 CFS 没意义（RT 才用）

2️⃣ 修改为 RT（⚠️ 慎用）

🔹 改为 FIFO（强实时）

  • -f = SCHED_FIFO
  • 20 = RT priority（1–99）

⚠️ 风险极高：

  • 不主动 yield 会霸占 CPU
  • 可直接卡死系统

🔹 改为 RR（可轮转）

  • 更安全
  • 适合周期性内核线程（如网络）

3️⃣ 只“稍微扶一把”：nice 而不是 RT

🔹 推荐方案（ksoftirqd 场景）

效果：

  • 仍是 CFS
  • 但 vruntime 增长更慢
  • 显著降低被饿死概率

👉 这是生产环境首选

4️⃣ 一次性批量调整 ksoftirqd

📌 通常 -5 已足够
📌 不建议 < -10

调度相关的其他关键观测工具

1️⃣ /proc/<pid>/sched

你能看到：

  • vruntime
  • nr_switches
  • se.exec_start

👉 这是理解“为什么被饿死”的核心文件

2️⃣ schedstat（系统级）

适合：

  • NUMA / 大核系统
  • 分析 runqueue 压力

3️⃣ perf sched

用途：

  • 精确看到谁在抢 CPU
  • 定位调度延迟根因

4️⃣ eBPF（你已经在用的最高阶）

  • sched:sched_switch
  • irq:softirq_raise
  • irq:softirq_entry

👉 你现在做的事情，已经是调度问题的最深水区

五、结合你当前问题的结论性建议

✅ 关于 ksoftirqd：

  • 不是随机 CPU

  • 是 per-cpu 绑定线程

  • 被饿死 ≠ 优先级低，而是：

    • CFS 公平性
    • 与业务进程同级竞争

✅ 推荐干预顺序（控制变量）：

  1. ❌ 不启 RPS
  2. ✅ renice -5 ksoftirqd/*
  3. ✅ IRQ 绑核（同 NUMA）
  4. ⛔ 谨慎使用 chrt
