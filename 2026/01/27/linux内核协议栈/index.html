协议栈

🧠 主线口诀

① 第一条主线：数据方向（最重要）

只保留 90% 正常流量会经过的路径,忽略异常、重传、边角状态机

TX：用户 → TCP → IP → 设备 → 网卡
RX：网卡 → IP → TCP → Socket → 用户

发送路径（左半边为主）

  • write / send / sendmsg
  • tcp_sendmsg
  • tcp_push_one
  • ip_output
  • dev_queue_xmit
  • driver → NIC

接收路径（右半边为主）

  • NIC 中断
  • NAPI poll
  • netif_receive_skb
  • ip_rcv
  • tcp_v4_rcv
  • sock_recvmsg / read

第二条主线：控制面 vs 数据面

图中混杂了两类东西，你要强行把它们分开：

🟨 数据面（你 80% 关注的）

  • skb（sk_buff）
  • 队列（queue）
  • copy / zero-copy
  • push / pull

关键词：

🟥 控制面（暂时只识别，不深挖）

  • socket 状态机
  • TCP 状态（ESTABLISHED / TIME_WAIT）
  • ACK / 重传 / 定时器

关键词：

二、工程上该关心什么

① User / Syscall 层

你关心什么：
• 应用是否阻塞？
• write/send 是不是慢？
• 是否频繁系统调用？

常见问题：
• 小包 write（syscall 过多）
• 未开启 sendfile / splice

⸻

② TCP 层（tcp_sendmsg / tcp_v4_rcv）

你关心什么：
• 队列是否满？
• 是否在重传？
• 延迟是否在 TCP 内部？

指标 / 工具：

ss -ti
netstat -s

⸻

③ IP 层（ip_output / ip_rcv）

你关心什么：
• 路由是否正确？
• 是否被 netfilter / iptables 拖慢？
• MTU / 分片问题？

典型场景：
• kube-proxy iptables 模式
• 大量规则导致 ip_rcv 慢

⸻

④ Qdisc / dev_queue_xmit（性能高发区）

这是工程事故高发区

你关心：
• 队列是否堆积？
• qdisc 是否成为瓶颈？
• 网卡 TX 是否 backpressure？

工具：

tc -s qdisc

⸻

⑤ NAPI / softirq（RX 性能关键）

这是 RX 侧最重要的地方

你关心：
• softirq 是否打满 CPU？
• backlog 是否丢包？
• 是否进入 ksoftirqd？

工具：

cat /proc/net/softnet_stat
top
perf top

三、工程级「怎么用这张图」

场景 1：QPS 上不去，但 CPU 很高

直接套路径：

RX → NAPI → netif_receive_skb → tcp_v4_rcv

优先看：
• softirq
• backlog
• TCP 接收队列

⸻

场景 2：延迟高，但没有丢包

优先怀疑：

TCP → 队列 → 调度

排查：
• TCP send/recv queue
• Nagle / delayed ACK
• cgroup CPU 限制

⸻

场景 3：Ingress / Service 性能异常

路径映射：

NIC
→ Linux RX
→ iptables / conntrack
→ Pod veth
→ 容器 TCP

重点：
• netfilter
• conntrack
• veth + qdisc
