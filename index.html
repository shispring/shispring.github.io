设计模式快速对比指南

发表于 2026-02-09

设计模式快速对比指南

创建型模式 (Creational Patterns)

模式 核心理念 解决问题 使用时机 关键词
工厂方法
(Factory Method)		 定义创建对象的接口，让子类决定实例化哪个类 对象创建的解耦，避免客户端依赖具体类 需要根据条件创建不同类型的对象时 “创建什么”
抽象工厂
(Abstract Factory)		 提供创建一系列相关或相互依赖对象的接口 创建产品族，保证产品族的一致性 需要创建多个相关产品族时 “产品族”
单例
(Singleton)		 确保一个类只有一个实例，并提供全局访问点 控制实例数量，提供全局访问点 需要全局唯一实例时 “唯一实例”
建造者
(Builder)		 将复杂对象的构建与表示分离 构建复杂对象，支持不同的表示 需要构建复杂对象，且构建过程需要灵活控制时 “分步构建”
原型
(Prototype)		 通过复制现有对象创建新对象 避免创建对象的开销，支持动态创建 对象创建成本高，或需要动态创建对象时 “克隆”

结构型模式 (Structural Patterns)

模式 核心理念 解决问题 使用时机 关键词
适配器
(Adapter)		 将一个类的接口转换成客户希望的另一个接口 接口不兼容，需要协同工作 需要集成接口不兼容的第三方库或遗留系统时 “接口转换”
装饰器
(Decorator)		 动态地给对象添加额外职责 动态扩展功能，避免类爆炸 需要动态地给对象添加功能，且功能可以组合时 “动态添加”
代理
(Proxy)		 为其他对象提供一种代理以控制对这个对象的访问 控制对对象的访问，延迟加载 需要控制对对象的访问，或延迟加载时 “访问控制”
外观
(Facade)		 为子系统中的一组接口提供一个统一的高层接口 简化复杂系统的接口 需要简化复杂子系统的接口时 “简化接口”
桥接
(Bridge)		 将抽象部分与实现部分分离，使它们可以独立变化 抽象与实现分离，避免永久绑定 需要在多个维度上独立扩展时 “分离抽象与实现”
组合
(Composite)		 将对象组合成树形结构以表示”部分-整体”的层次结构 统一处理单个对象和组合对象 需要表示对象的部分-整体层次结构时 “树形结构”
享元
(Flyweight)		 运用共享技术有效地支持大量细粒度的对象 减少对象数量，节省内存 需要大量相似对象，且可以共享内部状态时 “共享”

行为型模式 (Behavioral Patterns)

模式 核心理念 解决问题 使用时机 关键词
策略
(Strategy)		 定义一系列算法，把它们封装起来，并使它们可相互替换 算法封装与切换，避免条件语句 需要在运行时动态选择算法或行为时 “怎么做”
观察者
(Observer)		 定义对象间的一种一对多的依赖关系 状态变化通知，松耦合 一个对象的状态改变需要通知其他对象时 “通知”
模板方法
(Template Method)		 定义算法的骨架，将某些步骤延迟到子类 算法骨架不变，步骤可变 需要定义算法骨架，某些步骤由子类实现时 “算法骨架”
命令
(Command)		 将请求封装成对象，从而可以用不同的请求对客户进行参数化 请求封装，支持撤销/重做 需要将请求调用者与接收者解耦时 “请求封装”
责任链
(Chain of Responsibility)		 将请求沿着处理链传递，直到有对象处理它 请求处理，避免发送者与接收者耦合 多个对象可以处理请求，但不确定由谁处理时 “链式处理”
状态
(State)		 允许一个对象在其内部状态改变时改变它的行为 状态相关行为，避免大量条件语句 对象的行为依赖于它的状态，且状态会改变时 “状态驱动”
中介者
(Mediator)		 定义一个中介对象来封装一系列对象的交互 对象间交互的解耦，减少耦合 多个对象间存在复杂的交互关系时 “中介协调”
迭代器
(Iterator)		 提供一种方法顺序访问一个聚合对象中的各个元素 统一遍历聚合对象，不暴露内部表示 需要遍历聚合对象，且不暴露其内部表示时 “遍历”
访问者
(Visitor)		 表示一个作用于某对象结构中的各元素的操作 操作与对象结构分离，易于添加新操作 需要对一个对象结构中的对象进行多种不同的操作时 “操作分离”
备忘录
(Memento)		 在不破坏封装性的前提下，捕获一个对象的内部状态 状态保存与恢复，支持撤销 需要保存和恢复对象的状态时 “状态保存”
解释器
(Interpreter)		 给定一个语言，定义它的文法的一种表示，并定义一个解释器 解释语言，构建解释器 需要解释一种简单的语言时 “解释”

快速选择指南

根据问题类型选择

创建对象相关

  • 需要根据条件创建不同对象 → 工厂方法
  • 需要创建相关产品族 → 抽象工厂
  • 需要全局唯一实例 → 单例
  • 需要构建复杂对象 → 建造者
  • 需要复制对象 → 原型

接口与结构相关

  • 接口不兼容 → 适配器
  • 需要动态添加功能 → 装饰器
  • 需要控制访问 → 代理
  • 需要简化接口 → 外观
  • 需要在多个维度扩展 → 桥接
  • 需要树形结构 → 组合
  • 需要共享对象 → 享元

行为与交互相关

  • 需要动态选择算法 → 策略
  • 需要状态变化通知 → 观察者
  • 需要算法骨架 → 模板方法
  • 需要请求封装 → 命令
  • 需要链式处理 → 责任链
  • 需要状态驱动行为 → 状态
  • 需要解耦交互 → 中介者
  • 需要遍历对象 → 迭代器
  • 需要分离操作 → 访问者
  • 需要保存状态 → 备忘录
  • 需要解释语言 → 解释器

根据设计原则选择

开闭原则（对扩展开放，对修改关闭）

  • 策略模式：通过添加新策略扩展
  • 工厂模式：通过添加新工厂扩展
  • 装饰器模式：通过添加新装饰器扩展
  • 观察者模式：通过添加新观察者扩展

单一职责原则

  • 单例模式：只负责创建和管理唯一实例
  • 外观模式：只负责简化接口
  • 适配器模式：只负责接口转换

依赖倒置原则

  • 工厂模式：依赖抽象而非具体实现
  • 策略模式：依赖策略接口
  • 桥接模式：依赖抽象而非具体实现

常见组合使用

1. 工厂 + 策略

2. 装饰器 + 适配器

3. 观察者 + 单例

4. 工厂 + 单例

5. 责任链 + 命令

实际项目中的应用示例

Web 应用开发

  • 工厂模式：创建不同类型的控制器
  • 策略模式：不同的认证策略（JWT、OAuth、Session）
  • 装饰器模式：中间件（日志、认证、缓存）
  • 观察者模式：事件系统
  • 单例模式：数据库连接池、配置管理器

数据处理系统

  • 建造者模式：构建复杂的数据查询
  • 模板方法模式：数据处理流程框架
  • 责任链模式：数据验证链
  • 策略模式：不同的数据转换策略
  • 适配器模式：适配不同的数据源

游戏开发

  • 状态模式：角色状态（站立、行走、攻击）
  • 命令模式：玩家操作（移动、攻击、技能）
  • 观察者模式：游戏事件通知
  • 工厂模式：创建不同类型的游戏对象
  • 组合模式：游戏场景树

云原生应用

  • 策略模式：不同的云厂商适配
  • 适配器模式：统一不同云服务的接口
  • 工厂模式：创建不同类型的资源
  • 观察者模式：Kubernetes Watch 机制
  • 单例模式：客户端连接管理

避免过度设计

不需要设计模式的场景

  1. 代码简单，逻辑清晰
  2. 需求明确，不会频繁变化
  3. 性能要求极高，设计模式会带来额外开销
  4. 团队成员不熟悉设计模式

何时引入设计模式

  1. 代码中出现大量重复逻辑
  2. 使用大量 if-else 或 switch-case
  3. 类之间耦合严重，难以测试
  4. 需求频繁变化，代码难以维护
  5. 需要集成第三方库或遗留系统

学习建议

初学者

  1. 先理解最常用的5个模式：单例、工厂、策略、观察者、装饰器
  2. 在实际项目中尝试应用
  3. 阅读优秀开源项目的源码

进阶者

  1. 学习所有23种设计模式
  2. 理解模式之间的关系和组合使用
  3. 根据项目需求灵活选择和组合模式

高级者

  1. 深入理解设计原则
  2. 能够根据实际情况创造性地应用模式
  3. 能够识别反模式，避免过度设计

总结

设计模式不是银弹，而是解决特定问题的工具。关键在于：

  1. 理解本质：不要死记硬背，要理解每种模式解决的核心问题
  2. 从需求出发：根据实际要解决的问题选择合适的模式
  3. 避免过度设计：简单问题不要引入复杂模式
  4. 灵活应用：在复杂场景中，多种模式可以组合使用

记住：最好的设计模式是那些让代码更清晰、更易维护、更易扩展的模式。

controller-runtime中Builder模式设计

发表于 2026-02-04

1. Builder 结构体属性分析

2. 核心方法解释

For(object client.Object, opts ...ForOption)

  • 作用：定义 Controller 主要负责的资源类型。
  • 底层逻辑：相当于调用 Watches(&source.Kind{Type: apiType}, &handler.EnqueueRequestForObject{})。
  • 意义：告诉 Controller，当这个资源发生变化（增删改）时，触发 Reconcile 逻辑，且 Request 对象就是该资源的 NamespacedName。

Owns(object client.Object, opts ...OwnsOption)

  • 作用：定义 Controller 拥有的子资源类型。
  • 底层逻辑：相当于调用 Watches(&source.Kind{Type: <ForType>}, &handler.EnqueueRequestForOwner{OwnerType: apiType, IsController: true})。
  • 意义：当子资源（如 Pod）发生变化时，Controller 会通过 OwnerReference 找到它的”主人”（如 ReplicaSet），并触发”主人”的 Reconcile。这是 K8s 控制器模式的核心之一。

Watches(src source.Source, eventhandler handler.EventHandler, opts ...WatchesOption)

  • 作用：定义监控任意资源的通用方法。
  • 底层逻辑：最底层的监控注册接口。
  • 意义：处理复杂场景。例如，当 ConfigMap 变化时，你可能想 Reconcile 使用了该 ConfigMap 的 Deployment。这时 For 和 Owns 都不适用，需要用 Watches 配合自定义的 EventHandler（如 EnqueueRequestsFromMapFunc）来实现。

Complete(r reconcile.Reconciler)

  • 作用：完成构建过程。
  • 逻辑：
    1. 调用 doController：初始化 Controller 实例，设置并发数、日志等。
    2. 调用 doWatch：将 For、Owns、Watches 中定义的监控规则注册到 Controller 中。
    3. 返回构建好的 Controller。

3. 设计的好处

简化复杂性 (Abstraction)

  • 原始方式：如果不使用 Builder，你需要手动创建 Controller，手动实例化 Source、Handler、Predicate，并调用 ctrl.Watch。这需要对底层组件非常熟悉，代码量大且容易出错。
  • Builder方式：通过 For 和 Owns 这种语义化的 API，开发者只需声明”我关心什么资源”和”我拥有什么资源”，底层复杂的 Watch 注册逻辑被封装起来了。

声明式编程 (Declarative)

  • Builder 模式允许开发者以声明式的方式配置 Controller，而不是编写一系列命令式的初始化代码。代码可读性更强，意图更清晰。
    • 例如：NewControllerManagedBy(mgr).For(&Deployment{}).Owns(&ReplicaSet{}).Complete(r) 就像一句话一样清晰。

最佳实践封装 (Best Practices)

  • Builder 默认集成了 K8s 社区的最佳实践。
    • 例如，Owns 自动配置了 EnqueueRequestForOwner，这是处理 OwnerReference 的标准方式。
    • For 自动配置了 EnqueueRequestForObject，这是 Reconcile 自身资源的标准方式。

灵活性与扩展性 (Flexibility)

  • 虽然提供了 For 和 Owns 这种便捷方法，但也保留了 Watches 和 WithOptions 等底层接口，允许高级用户处理复杂的边缘情况（如监控外部事件、自定义队列处理逻辑等）。

依赖注入 (Dependency Injection)

  • 通过 ControllerManagedBy(mgr) 传入 Manager，Builder 可以自动获取所需的 Client、Scheme、Logger 等依赖，无需开发者手动传递，降低了组件间的耦合度。

限流算法

算法名称 原理一句话 优点 缺点 典型案例 / 场景
计数器（Counter） 累加请求数，超过阈值拒绝 实现极简单 无时间维度；洪峰瞬间失效 Demo / 教学
固定窗口（Fixed Window） 每个时间窗口内最多 N 次 简单、易理解 窗口边界突刺 简单接口限流
滑动窗口（Sliding Window） 统计最近 T 时间内请求数 平滑、精确 实现复杂、存储成本高 SDK / API 网关
漏桶（Leaky Bucket） 请求进桶，固定速率漏出 输出稳定、可整形 不支持突发流量 网络整形、QoS
⭐ 令牌桶（Token Bucket） 定速生成令牌，请求消耗 支持突发、弹性好 实现略复杂 生产主流方案

限流 ≠ 平滑
限流 = 允许爆发，但要兜底

协议栈

🧠 主线口诀

① 第一条主线：数据方向（最重要）

只保留 90% 正常流量会经过的路径,忽略异常、重传、边角状态机

TX：用户 → TCP → IP → 设备 → 网卡
RX：网卡 → IP → TCP → Socket → 用户

发送路径（左半边为主）

  • write / send / sendmsg
  • tcp_sendmsg
  • tcp_push_one
  • ip_output
  • dev_queue_xmit
  • driver → NIC

接收路径（右半边为主）

  • NIC 中断
  • NAPI poll
  • netif_receive_skb
  • ip_rcv
  • tcp_v4_rcv
  • sock_recvmsg / read

第二条主线：控制面 vs 数据面

图中混杂了两类东西，你要强行把它们分开：

🟨 数据面（你 80% 关注的）

  • skb（sk_buff）
  • 队列（queue）
  • copy / zero-copy
  • push / pull

关键词：

🟥 控制面（暂时只识别，不深挖）

  • socket 状态机
  • TCP 状态（ESTABLISHED / TIME_WAIT）
  • ACK / 重传 / 定时器

关键词：

二、工程上该关心什么

① User / Syscall 层

你关心什么：
• 应用是否阻塞？
• write/send 是不是慢？
• 是否频繁系统调用？

常见问题：
• 小包 write（syscall 过多）
• 未开启 sendfile / splice

⸻

② TCP 层（tcp_sendmsg / tcp_v4_rcv）

你关心什么：
• 队列是否满？
• 是否在重传？
• 延迟是否在 TCP 内部？

指标 / 工具：

ss -ti
netstat -s

⸻

③ IP 层（ip_output / ip_rcv）

你关心什么：
• 路由是否正确？
• 是否被 netfilter / iptables 拖慢？
• MTU / 分片问题？

典型场景：
• kube-proxy iptables 模式
• 大量规则导致 ip_rcv 慢

⸻

④ Qdisc / dev_queue_xmit（性能高发区）

这是工程事故高发区

你关心：
• 队列是否堆积？
• qdisc 是否成为瓶颈？
• 网卡 TX 是否 backpressure？

工具：

tc -s qdisc

⸻

⑤ NAPI / softirq（RX 性能关键）

这是 RX 侧最重要的地方

你关心：
• softirq 是否打满 CPU？
• backlog 是否丢包？
• 是否进入 ksoftirqd？

工具：

cat /proc/net/softnet_stat
top
perf top

三、工程级「怎么用这张图」

场景 1：QPS 上不去，但 CPU 很高

直接套路径：

RX → NAPI → netif_receive_skb → tcp_v4_rcv

优先看：
• softirq
• backlog
• TCP 接收队列

⸻

场景 2：延迟高，但没有丢包

优先怀疑：

TCP → 队列 → 调度

排查：
• TCP send/recv queue
• Nagle / delayed ACK
• cgroup CPU 限制

⸻

路由Demo解析

发表于 2026-01-27

时序流程图（Mermaid）

一、“总纲式结论”

跨三层通信的本质是：

  • IP 地址：端到端不变（逻辑寻址）
  • MAC 地址：逐跳改变（物理转发）
  • ARP：只解决“下一跳是谁”，而不是“最终是谁”

二、拓扑与角色明确

拓扑如下（你给的）：

  • Host1 的默认网关：Router A
  • Host2 直连在 Router C 的 LAN 上
  • 路由器之间都是三层转发
  • 终端和路由器之间是二层通信

三、完整路由过程（按步骤拆解）

阶段 1：Host1 决定“我该把包交给谁”

Step 1：Host1 判断目的 IP 是否本地

  • Host1 拿到 Host2 的 IP

  • 用自己的：

    • IP
    • 子网掩码

  • 计算后发现：
    Host2 不在本地子网

👉 结论：必须交给默认网关（Router A）

Step 2：Host1 通过 ARP 找默认网关的 MAC

  • 查看 ARP 表：

    • 有 Router A MAC → 直接用
    • 没有 → ARP Broadcast

  • 得到：

    • Router A（LAN 口）的 MAC

Step 3：Host1 封装第一个以太网帧

此时数据包状态是：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Host1 MAC
Dest MAC Router A（LAN 口）MAC

👉 关键点：
Host1 不知道 Router B / C / Host2 的 MAC，也不需要知道。

阶段 2：Router A → Router B

Step 4：Router A 收到帧（L2 → L3）

Router A 做三件事：

  1. 看 目的 MAC → 是自己 → 收包

  2. 剥掉二层头，检查 目的 IP

  3. 查路由表：

    • Host2 不直连
    • 找到下一跳：Router B

Step 5：Router A 通过 ARP 找 Router B 的 MAC

  • ARP 查询：

    • IP：Router B（A-B 链路上的 IP）
    • 得到 Router B 接口 MAC

Step 6：Router A 重新封装帧并转发

新的帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router A（发往 B 的接口 MAC）
Dest MAC Router B（面向 A 的接口 MAC）

👉 IP 没变，MAC 换了

阶段 3：Router B → Router C（同理）

Router B 做的事情和 Router A 完全一样：

  1. 收到帧（MAC 是自己）
  2. 解封装
  3. 查路由表
  4. 下一跳是 Router C
  5. ARP 找 Router C MAC
  6. 重新封装并转发

新的帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router B（面向 C 的接口 MAC）
Dest MAC Router C（面向 B 的接口 MAC）

阶段 4：Router C → Host2（最后一跳）

Step 7：Router C 发现目的 IP 是本地直连

  • 查路由表：

    • Host2 IP 在本地 LAN

  • 不再需要“下一跳路由器”

Step 8：Router C ARP 查询 Host2 的 MAC

  • ARP：

    • IP：Host2
    • 得到 Host2 MAC

Step 9：Router C 发送最终帧

最终帧：

字段 值
Source IP Host1 IP
Dest IP Host2 IP
Source MAC Router C（LAN 口 MAC）
Dest MAC Host2 MAC

Step 10：交换机（二层）转发

  • 交换机只看 Destination MAC
  • 查 MAC 表
  • 发给 Host2 所在端口

👉 Host2 收到数据，去程完成

四、一路上哪些东西「一直不变」，哪些「一直在变」

✅ 一直不变的

  • Source IP：Host1
  • Destination IP：Host2

🔄 每一跳都会变的

  • Source MAC
  • Destination MAC

🧠 决策逻辑

  • IP 决定“去哪里”
  • MAC 决定“下一跳交给谁”

五、为什么不能“全网 ARP”（你原文最后的点睛）

如果没有路由器：

  • 所有设备在一个广播域
  • 每一次通信都 ARP 全网
  • 广播风暴
  • 网络直接瘫痪

👉 路由器的核心价值：

  • 切割广播域
  • 限制 ARP 的作用范围
  • 让 Internet 成为可能

六、一句“工程师级总结”

三层决定方向，二层负责搬运。
路由器每一跳只关心：
“下一步我该把包交给谁？”

一、分析图片的 3 个步骤

Step 1：给 Pod 插上“网线”（veth）

创建 veth pair，一端进 Pod，一端留在 Node

👍 这是 CNI 的核心职责之一

底层原理：

  • veth 是 Linux 提供的“虚拟网线”
  • Pod 里看到的是 eth0
  • Node 上看到的是 vethxxxx
    👉 CNI 插件最基础的职责：

    把 Pod 的 network namespace 接入宿主机网络

Step 2：给 Pod 分配 IP（IPAM）

每个 Pod 一个唯一 IP，通常 Node 拿一个子网

👍 这也是 CNI 的标准设计

这是：
* Node 级子网划分
* Pod 从 Node 子网里分 IP
👉 这是非常典型的 Node-scoped IPAM 模型
Flannel / Calico / Cilium 都支持这个模型

Step 3：配置 IP + 路由（L3 通）

Pod 能出网，Node 知道怎么把包送回 Pod

👍 这是最容易被低估，但最重要的一步

图里提到三件事：
1. 给 Pod eth0 配 IP
2. 给 Pod 配默认路由
3. 给 Node 配到 Pod IP 的路由
这是 “Pod 能通信”成立的最小条件。
