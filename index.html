容器文件系统挂载

发表于 2025-09-19

1. 查看挂载情况

  • 看挂载情况

  • 看是否有进程占用

    或者：

  • 看 containerd 里是否有残留 metadata

  1. ctr -n k8s.io tasks list | grep 200ab1146de → 状态 STOPPED
  2. fuser -vm 显示很多 containerd-shim、qemu-system-x86、virtiofsd 等进程引用这个目录
    • 说明这个目录（mountpoint）仍然被多个进程占用着。
    • 注意：这里的 “F.c..” / “frc..” 表示打开了这个目录的 文件描述符，并且有读写/控制引用。
  3. 整体情况
    • 内核 overlayfs 挂载点还在，inode 没回收。
    • 容器已经 “STOPPED”（不可用），但是挂载点被其他 shim / qemu / virtiofsd 引用，所以无法自动卸载。

2. 命令：

这个命令会做什么？

  • 找出所有挂载在 /run/containerd/io.containerd.runtime.v2.task/k8s.io/* 下的 mount point。
  • 对它们执行 umount -l（lazy umount）。

lazy umount 含义：

  • 马上把挂载点从 mount namespace 里“摘掉”，进程不再能通过路径访问它。
  • 如果有进程仍然持有 fd，会等它们关闭 fd 之后再真正释放资源（inode、dentry、slab）。
  • 这样避免了 umount 报错 “busy”，算是比较温和的方式。

3. 对当前系统的影响

  • 如果容器仍然活跃：对应 Pod 会立刻失效，因为 rootfs 被摘掉了，里面的文件系统消失。

  • 如果容器已经 STOPPED（像你这种）：

    • 大多数情况下影响有限，等价于清理垃圾挂载点。
    • 不过要小心：如果 containerd metadata 里还记录这个容器存在，containerd 在后续 GC 或者回收时可能会发现找不到挂载点，从而报错。

4. 建议操作顺序

更温和的方案可以这样分级：

  1. 优先让 containerd 清理

    （如果卡住，说明 shim/进程依赖还在）

  2. 确认 mount

  3. 再 lazy umount

  4. 最后清理目录

  5. 如果有大量残留，你那个 awk | xargs umount -l 可以作为批量清理手段，但最好在确认这些容器都已经 STOPPED 或已被 K8s 删除的前提下做。

✅ 结论：

  • 你的 umount -l 是温和可行的，不会直接 kill 进程，但会让容器 rootfs 消失。
  • 如果只是清理 STOPPED 的残留容器，几乎没副作用，还能回收 inode/slab。
  • 建议 先用 ctr rm 尝试让 containerd 自己清理，再手动 umount + rm 兜底。

cgroup/namespace vs Kata

发表于 2025-09-05

概念快速梳理

  • namespace (PID/NET/IPC/UTS/MNT/USER/…) ：Linux 提供的进程隔离机制（名字空间），用于隔离进程视角（PID、网络命名空间等）。
  • cgroup (control groups) ：Linux 提供的资源控制（CPU/memory/io/pids），用于限制与计量资源消耗。
  • 传统容器 (Docker/runc etc.) ：基于单一宿主机内核 + namespaces + cgroups，为应用提供轻量隔离。
  • Kata Containers：为容器提供轻量级虚拟化：每个容器或 Pod 运行在一个轻量 VM（有独立内核），兼顾 VM 的强隔离与容器的速度。

关键差别（对比表）

维度 Namespaces + cgroups (传统容器) Kata Containers (轻量 VM)
隔离级别 进程级隔离，同一宿主机内核共享 每个 VM 有自己内核（更强的隔离）
内核 bug 影响 宿主机内核 bug 可能影响所有容器 客户 VM 内核问题通常仅影响该 VM（保护宿主）
启动速度 极快（ms ~ s） 较慢（s 量级，取决于 runtime/VM）
资源开销 很低（共享内核，少量 overhead） 较高（每 VM 有 qemu/kvm 等开销）
安全边界 用户态较弱（容器逃逸有历史） 更强（硬隔离，适合多租户/不信任工作负载）
调试与运维 与宿主一致（/proc, cgroup） VM 内/宿主 双层调试，复杂度高
对单 Pod 内存泄漏影响 泄漏在 host kernel/user space，可直接影响 host 泄漏限制在 VM（若VM有 memory limit 则受限）
使用场景 大多数云原生应用、CI/CD、低延迟服务 多租户且对安全敏感、需要防止内核级风险或特定内核版本隔离的场景

Kata 能否避免单业务导致全主机挂掉？

  • 能显著降低风险：如果问题是内核级 bug 或需要特殊内核模块出问题（例如某些驱动、netfilter bug），在 Kata 中发生时通常只影响该 VM，不会直接影响宿主机内核和其他 VM/容器。
  • 对用户空间内存泄漏：如果单个 VM 的进程泄漏内存并且该 VM 的内存上限被设置（例如给 VM 分配 4Gi），则泄漏会耗尽该 VM 内存并导致 VM 内 OOM，但不会直接耗尽宿主整台机器的内存（只要宿主上没有被配置过大的 VM 数量）。
  • 注意：大量 Kata VM 被同时启动且每个都泄漏，仍然能用尽宿主资源 —— 因此仍需上层资源限制 & 调度策略。

什么时候优先考虑使用 Kata？

  • 多租户平台（每租户代码不信任），需要强内核隔离。
  • 运行需要自定义/不同内核版本或特殊内核模块的 workload，避免宿主内核污染。
  • 法规或安全要求必须严格隔离（例如金融 / 医疗 / 隐私敏感场景）。
  • 需要兼顾容器化便捷性与 VM 级安全边界。

运维与排查差异

  • 传统容器：
    • 优势：直接查看宿主 /sys/fs/cgroup、/proc 来诊断 container 内资源使用（更直观）。
    • 常用工具：docker ps/ctr、systemd-cgls、cat cgroup memory.stat。
  • Kata：
    • 你需要在宿主层和Guest VM 层都排查：宿主层看 VM 进程、资源分配，guest 内看应用行为。
    • 更复杂：两层日志、两层指标，但隔离性更好。

实践建议（基于场景）

  1. 大多数应用：先用传统容器 + 严格 cgroup / kubelet eviction /资源限额。
  2. 不信任租户 / 强隔离：用 Kata（或 VM-based multi-tenant）并为每 VM 设置合理内存上限。
  3. IO / 存储敏感：优化后端存储、设置 vm.dirty_* 限额、在 runtime 层配合 QoS 限制。
  4. 混合部署：把高风险工作负载放 Kata，普通服务放容器，减少总体成本。

结语

  • cgroup + namespace 提供高效、低开销的多租户能力，适合绝大多数云原生场景；
  • Kata 在强隔离与安全场景下胜出，但有成本与运维复杂度，需权衡。

内存回写/脏页/回收/OOM

发表于 2025-09-05

内存回写 / 脏页 / 回收 / OOM — 全流程说明与排查指南

概览（一句话）

用户进程写数据 → 数据先写入页缓存（page cache）成为 dirty pages → 内核异步回写（writeback）到磁盘 → 若磁盘/后端慢或写入速率低于生成速率，dirty 将积累 → 内存回收（kswapd）尝试回收或触发写回并阻塞写者 → 若回收失败并耗尽内存，OOM killer 根据策略选择并杀死进程 / cgroup。

一、关键概念（快速回顾）

  • **页缓存 (page cache)**：文件/块 IO 的缓存层，存储在内存中 struct page。
  • Dirty pages：已修改但尚未写回到磁盘的页（占用内存，可写回）。
  • Writeback：内核将 dirty page 写回存储（由 writeback kthreads / flush workers 执行）。
  • kswapd / page reclaim：当内存紧张时回收可回收页（包括触发 writeback）。
  • OOM killer：当内核无法满足分配且无法回收，选择进程作为牺牲者（基于 oom_score、oom_score_adj、cgroup）并终止之。
  • cgroup v1/v2：容器资源限制机制，内存超限会触发 cgroup OOM 行为。

二、典型触发路径（更详细）

  1. 用户进程 write()

    • 内核在 page cache 分配/获取 page，复制数据到 page，然后标记为 dirty。
    • write 返回快（写入内存），不等磁盘写完 -> 写入吞吐高但延迟低。

  2. 后台 writeback

    • 内核周期性/阈值触发 writeback（vm.dirty_background_ratio / vm.dirty_ratio 或绝对量 vm.dirty_bytes）。
    • writeback 由内核 kthreads 执行，异步把页写到块设备或通过 NFS/CIFS/CSI 等上游。

  3. 当 writeback 跟不上

    • 若产生 dirty 的速度 > 写回速度（例如磁盘慢/IO 卡住），dirty 会累积。
    • 一旦达到 vm.dirty_ratio，写者可能被强制阻塞（同步地 flush），导致大量进程阻塞在 D (uninterruptible sleep) 状态。
    • IO 长时间阻塞会使系统响应变差（ssh 卡、syscalls 卡）。

  4. 内存回收与 OOM

    • kswapd/直接分配者触发回收，回收优先回收可回收的缓存页或回写脏页。
    • 如果回收仍然不能满足内存请求，内核触发 OOM killer，选择最合适的 victim（参考 oom_score, oom_score_adj, oom_badness 以及 cgroup 权重），并杀掉进程。
    • 在 cgroup v2 下，若 cgroup 达到 memory.max，可能只杀该 cgroup 内的进程（cgroup OOM）。

三、和 buff/cache / MemAvailable 的常见误解

  • buff/cache 高 不等于 可用性低；buff/cache 是可回收的（通常）。
  • 但当脏页非常多或 slab（内核对象）泄漏时，内存是“不可回收”的 → 即便 MemAvailable 暂时显示不是 0，也可能回收失败导致假死。
  • 区分点：看 Dirty/Writeback 与 SReclaimable / SUnreclaim（/proc/meminfo 与 slabtop）。

四、实操排查清单（从轻到重）

1) 快速查看内存/脏页/回写

free -h
cat /proc/meminfo | egrep 'Dirty|Writeback|MemAvailable|Cached|Buffers|SUnreclaim|Slab|Swap'
cat /proc/vmstat | egrep 'pgfault|pgmajfault|pgscan|pgsteal|nr_dirty|nr_writeback'

Content-Type 类型与使用场景

发表于 2025-08-14

消息体的数据格式

类型 示例 使用场景
application/json Content-Type: application/json 传输结构化数据（API 常用，如 REST 接口）。
application/xml Content-Type: application/xml 传输 XML 格式数据（老系统或 SOAP 协议）。
text/plain Content-Type: text/plain; charset=utf-8 传输纯文本（调试、日志、简单字符串）。
text/html Content-Type: text/html; charset=utf-8 传输 HTML 文档（网页响应）。
application/x-www-form-urlencoded Content-Type: application/x-www-form-urlencoded 表单提交（HTML form 的默认提交格式）。
multipart/form-data Content-Type: multipart/form-data; boundary=----xxxx 文件上传（表单 + 文件混合提交）。
application/octet-stream Content-Type: application/octet-stream 二进制数据（文件下载、任意二进制传输）。
text/event-stream Content-Type: text/event-stream SSE（Server-Sent Events）长连接，服务器持续推送事件给客户端。
application/grpc Content-Type: application/grpc gRPC RPC 协议，基于 HTTP/2 的二进制传输。

SSE vs chunked JSON

特性 SSE (text/event-stream) JSON + chunked (K8s watch)
格式 data: 行，纯文本 每个 chunk 独立 JSON 对象
使用场景 前端实时推送 后端 API 实时变更通知
浏览器支持 原生支持（EventSource） 需要手动解析
优势 易读、简单 结构化数据、后端友好
缺点 数据结构简单 需要解析 JSON 流
特性 K8s watch (chunked JSON) SSE
Content-Type application/json text/event-stream
数据格式 JSON 自定义事件流（以 data: 开头）
保持方式 HTTP/1.1 长连接 + chunked HTTP/1.1 长连接 + 流式传输
客户端解析 JSON decoder SSE parser
应用场景 Kubernetes API 事件 浏览器端推送消息

keep-alive 类别

类型 层级 特征
TCP keep-alive TCP 协议层 操作系统内核发的探测包（仅几个字节），目的是探测连接是否还活着。
HTTP keep-alive HTTP 应用层 通过 Connection: keep-alive 维持 TCP 连接，后续请求共用同一 TCP 通道。

SSE 理论 vs K8s watch 的实际连接时长

  • SSE（text/event-stream）：理论上无限长，只要网络不断开，服务器不重启，就能一直推送。

  • K8s watch：

    • 实际受限于 --min-request-timeout 和 API Server / 反向代理 / LB 的空闲连接超时。

    • 常见原因导致断开：

      1. kube-apiserver 超时（默认 30 分钟）。
      2. API Server 节点重启或滚动升级。
      3. 前面有 nginx / ELB / F5 等，配置了 idle timeout（可能只有 60 秒、300 秒等）。
      4. 客户端自己设了超时（例如 kubectl 默认超时时间）。

    • 因此 K8s 客户端都会实现 自动重连（比如 client-go 会带上 resourceVersion 重连）。

k8s watch保持长连接机制

它的核心机制：

  1. HTTP/1.1 持久连接（Persistent Connection）

    • TCP 层连接不关闭。
    • HTTP 请求和响应在同一条 TCP 连接上传输。

  2. Chunked Transfer Encoding

    • 服务端不一次性返回全部数据，而是分多次“分块”推送。
    • 每个分块前会有一个 16 进制的长度标识。
    • 客户端读取到数据就处理，而不是等到连接结束。

  3. K8s watch 特性

    • API Server 接收到 watch=true 请求后，会一直监听资源变化。
    • 每当资源变化，就向这个连接推送一条 JSON。
    • 没有变化时，不会发数据（可能会有周期性心跳，取决于实现）。

  4. 连接保活

    • TCP 层可能会开启 keep-alive 探测（由 OS 配置，比如 Linux /proc/sys/net/ipv4/tcp_keepalive_*）。
    • API Server 层会通过 --min-request-timeout 决定最小存活时长（默认 30 分钟）。
    • 客户端（如 kubectl / client-go）会自动在断开后重连。
